## Слайд 4 — Исходная архитектура (проблема)

**Макет:** Заголовок + диаграмма архитектуры (изображение) + список проблем справа.

**Содержимое:**

**Описание исходного подхода:**
- Pod = 2 контейнера: `storage-initializer` + `runtime-container`
- `storage-initializer` скачивает веса из HuggingFace в `emptyDir`
- После завершения инициализации runtime загружает модель в GPU

**Проблемы (список с иконками ❌):**
- ❌ Каждый под скачивает модель **заново** из интернета — 75 с для 15 ГБ
- ❌ При масштабировании до 10 реплик: **150 ГБ** внешнего трафика
- ❌ Нет переиспользования данных между подами и нодами
- ❌ Жёсткая привязка к доступности HuggingFace Hub
- ❌ `T_first_byte` = `T_model_ready` ≈ 75 с — инференс не может начаться раньше

**Визуал:** Использовать имеющуюся картинку архитектуры с `emptyDir`. Красные стрелки на проблемных местах.

## Слайд 5 — Требования к архитектуре

**Макет:** Заголовок + два блока (функциональные и нефункциональные требования).

**Функциональные требования:**
- Переиспользование весов между запусками (кэш на ноде)
- Передача данных между нодами без внешнего источника (P2P)
- Стандартный файловый интерфейс для ML-runtime (POSIX)
- Автоматический fallback на внешний репозиторий при отсутствии кэша
- Совместимость с Kubernetes (DaemonSet, PVC, CSI)

**Нефункциональные требования:**
- `T_model_ready` при local cache hit: ≤ 3 с (для 15 ГБ модели)
- `T_first_byte` (lazy loading): ≤ 1 с
- Внешний трафик при N=10 репликах: = 1× размер модели (не N×)
- Отказ одной worker-ноды не должен вызывать полную недоступность модели
- Overhead FUSE при последовательном чтении: ≤ 1% CPU
- Redis lookup latency: p99 < 5 мс

**Визуал:** Два столбца с иконками (функциональные — синие, нефункциональные — зелёные).

## Слайд 7 — Предложенная архитектура

**Макет:** Заголовок + большая диаграмма архитектуры (на весь слайд или 70% ширины) + описание компонентов справа/снизу.

**Три ключевых компонента:**

**1. storage-agent (DaemonSet)**
- Запускается на каждой worker-ноде
- Управляет локальным chunk-кэшем на NVMe-диске
- Обслуживает запросы от storage-mounter через Unix Domain Socket (UDS)
- Обменивается чанками с агентами на других нодах через TCP

**2. storage-mounter (FUSE-клиент)**
- Монтирует виртуальную файловую систему в pod
- ML-runtime видит обычные файлы — нет изменений в коде модели
- Lazy loading: первый байт доступен до полной загрузки модели

**3. Redis Cluster (Control Plane)**
- Хранит: какие чанки есть, на каких нодах, с каким TTL
- Агенты публикуют heartbeat каждые 10 с
- При отказе TTL истекает через 30 с → автоматическая очистка

**Алгоритм чтения чанка (3 уровня):**
```
1. Локальный кэш (NVMe) → 2.3 мс
      ↓ miss
2. Соседняя нода (P2P/TCP) → 13.3 мс
      ↓ miss
3. HuggingFace Hub (fallback) → 130+ мс
```

**Визуал:** Использовать имеющуюся картинку архитектуры P2P/FUSE-системы. Желательно в цвете с выделением трёх уровней.

## Слайд 10 — Анализ надёжности

**Макет:** Заголовок + три блока сценариев отказов.

**Сценарий 1: Отказ worker-ноды**
- TTL-записи агента истекают через 30 с
- Затронуты только чанки без репликации на других нодах
- При r=2: P(потери) = 10⁻⁴ → при r=3: P(потери) = 10⁻⁶
- **Результат:** деградация к HuggingFace fallback, не остановка

**Сценарий 2: Недоступность Redis**
- Потеря peer discovery — агент не видит соседей
- Система деградирует: local cache → HuggingFace (без P2P)
- **Результат:** сервис продолжает работу, замедление только при miss

**Сценарий 3: Недоступность HuggingFace Hub**
- Если чанк есть в local или peer cache — запрос обслуживается
- При полном отсутствии кэша — ошибка (как и у baseline)
- **Результат:** прогретый кластер полностью изолирован от внешнего источника

**Вывод:** NFS при любом отказе сервера — полная остановка. P2P/FUSE деградирует gracefully.

**Визуал:** Три карточки с иконками «нода», «Redis», «облако». Статусы: ⚠️ Деградация / ✅ Продолжает работу / ❌ Остановка.

> Система позволяет сократить задержку холодного старта на порядок при отсутствии деградации надёжности — что критично для production serverless ML inference.
