Описание картинки:
Архитектуру можно логически разделить на несколько ключевых уровней взаимодействия:
```
1. Уровень Control Plane (Реестр метаданных)
В качестве центрального источника правды (телефонной книги кластера) выступает сервис Redis, размещенный на выделенном узле api-node-1. Он выполняет роль легковесного P2P-трекера. Когда любой из фоновых воркеров (worker-daemon) успешно скачивает новый чанк модели, он регистрирует этот факт в Redis. Клиенты используют этот реестр для быстрого поиска метаданных: где именно (на каком узле и в каком объеме) лежат нужные части файлов.

2. Локальный Data Plane (Инференс и виртуальная ФС)
Внутри вычислительного узла (на схеме справа) запущен контейнер с ML-движком vllm и компонент mounter.

mounter — это FUSE-клиент, который монтирует виртуальную файловую систему в локальное хранилище пода (pod-storage). Для движка vllm создается полная иллюзия того, что гигантский файл модели уже целиком лежит на диске.

Когда vllm пытается прочитать определенный диапазон байтов (например, загрузить конкретный слой нейросети), mounter перехватывает этот системный вызов и по ультрабыстрому локальному каналу Unix Domain Socket (unix read) запрашивает эти данные у локального фоновго сервиса worker-daemon.

Локальный worker-daemon в первую очередь проверяет свое кэш-хранилище (localstorage) и, если данные присутствуют, мгновенно отдает их клиенту.

3. Горизонтальное масштабирование (P2P-сеть)
Если нужного куска данных на локальном диске не оказалось (Cache Miss), система не спешит обращаться к внешнему интернету. Опираясь на метаданные из Redis, клиент (или локальный демон) узнает, что нужный чанк уже есть у соседнего узла (на схеме слева). Происходит прямое обращение к соседнему worker-daemon по протоколу TCP, и данные передаются на скорости внутренней сети датацентра, минуя узкие места.

4. Внешний Fallback (Источник данных)
Если запрашиваемый чанк является абсолютно новым для всего кластера (его нет ни в локальном кэше, ни на соседних узлах в Redis), локальный worker-daemon обращается к первичному внешнему источнику — репозиторию HuggingFace (через прокси-сервер). Скачанный чанк размером в несколько мегабайт сохраняется в localstorage, отдается в vllm для вычислений, а информация о его появлении асинхронно публикуется в центральный Redis, делая его доступным для всех остальных узлов кластера.
```
