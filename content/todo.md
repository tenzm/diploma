Ориентировочный объем: **0.8-1 страница**.

- Serverless экономит ресурсы, когда нагрузка нерегулярна.

- Требуется механизм, который сохраняет преимущества serverless-масштабирования, но уменьшает повторную загрузку больших файлов модели при запуске новых реплик.

## 6. Раздел 3. Промежуточное решение на базе NFS и координации загрузок

### 6.1. Архитектура исходного решения `storage-initializer + emptyDir`

Ориентировочный объем: **1.5-2 страницы**.

Раскрыть:

- Каждый pod скачивает модель независимо.
- `emptyDir` живет в пределах pod/node lifecycle.
- Повторное скачивание при пересоздании pod на другой ноде.
- Слабое переиспользование данных между репликами.

Нужная схема:

- Pod с двумя контейнерами: `storage-initializer` и `runtime-container`.

### 6.2. Переход к NFS

Ориентировочный объем: **2-2.5 страницы**.

Использовать:

- `research/03_1_nfs_architecture.md`
- `research/03_nfs_synchronization_leader_election.md`

Раскрыть:

- NFS как общее RWX-хранилище.
- PersistentVolume/PersistentVolumeClaim.

### 7.4. Обнаружение узлов: Gossip, mDNS/UDP Broadcast, Redis heartbeat

Сравнить:

Gossip:

- Минусы: eventual consistency, несколько сетевых раундов, фоновый трафик, сложнее отлаживать.

Вывод:

- Для дипломной задачи, где cold start измеряется миллисекундами/секундами и требуется быстрый поиск чанков, Redis heartbeat предпочтительнее Gossip и mDNS.

### 7.6a. Кэш и буферизация чтения в storage-mounter: ring buffer, prefetcher, byte pool

Ориентировочный объем: **2-2.5 страницы**.

Описать реализованную в коде многоуровневую буферизацию:

- **Streaming ring buffer** (`internal/cache/streaming/`): кольцевой буфер фиксированной ёмкости, индекс `chunkId % capacity`. O(1) Get/Set. Верификация попадания по `chunk.Id == chunkId` (защита от коллизий). Оптимален для последовательного чтения одного файла.
- **Prefetcher** (`pkg/prefetcher/`): при каждом `Read(chunkId)` асинхронно запускает горутины для следующих `prefetchAhead-1` чанков. Использует `singleflight.Group` для исключения дублирующихся конкурентных загрузок одного чанка (thundering herd prevention). Параллелизм ограничивается `prefetchParallelism`.
- **Byte buffer pool** (`internal/pool/`): мультиразмерный `sync.Pool`. Для каждого уникального `chunkSize` — отдельный пул. Устраняет `make([]byte, chunkSize)` при каждом запросе, снижая нагрузку на GC.
- **In-memory metadata cache** (`internal/cache/metadata/`): unbounded map для `FileMetadata`, с поддержкой cascade delete по ревизии модели.

Раскрыть компромиссы:
- Ring buffer оптимален для sequential single-file паттерна (ML-загрузка весов); при multi-file (несколько .safetensors шардов) даёт коллизии → нужен per-file LRU.
- Prefetcher скрывает I/O latency за временем GPU-обработки текущего чанка.
- sync.Pool критичен при высоком параллелизме FUSE: без пула каждый syscall read() создаёт большой heap alloc.

### 7.8. Алгоритм чтения чанка

Ориентировочный объем: **3-4 страницы**.

Сценарий чтения:

1. `vLLM` вызывает `read()` по пути файла модели.
2. FUSE передает запрос в `storage-mounter`.
3. `storage-mounter` вычисляет `chunk_id` по `offset` и `chunk_size`.
4. `storage-mounter` отправляет запрос локальному `storage-agent` через UDS.
5. `storage-agent` проверяет localstorage.
6. При local cache hit возвращает данные сразу.
7. При local miss обращается к Redis за списком пиров.
8. Если пир найден, скачивает чанк по TCP у соседнего `storage-agent`.
9. Если пир не найден, скачивает из HuggingFace/другого registry.
10. Сохраняет чанк в localstorage.
11. Регистрирует чанк в Redis.
12. Возвращает байты в `storage-mounter`.
13. FUSE возвращает данные runtime-процессу.

Нужная диаграмма:

- Sequence diagram для cache hit, peer hit и remote fallback.

Формулы:

- AMAT: `AMAT = T_hit + (1 - H) * T_miss`.
- Расширенная модель:
  `E[T_read] = H_local*T_local + (1-H_local)*H_p2p*T_p2p + (1-H_local)*(1-H_p2p)*T_remote`.

Использовать:

- `theorems/01_amat.md`

- Redis должен быть развернут в HA-режиме.
- TTL защищает от stale metadata.
- External fallback сохраняет доступность даже при полном cache miss.

Формула:

- Доступность при репликации:
  `A = 1 - (1 - a)^r`.

Рисунок (вставлен):

- `result/figures/chart_availability.png` — простой в год (log-шкала) по коэффициенту репликации.
  Место в тексте: после описания органической репликации в `result/contents/4_1-p2p-fuse-architecture.tex`.

Вывод:

- В системе допускается eventual consistency для metadata о чанках, но чтение данных должно проверять фактическую доступность пира и иметь fallback.

### 7.12. Выводы по разделу

Ориентировочный объем: **1 страница**.

Главный вывод:

- Разработанная архитектура переносит хранение горячих данных ближе к compute-нодам, убирает NFS как центральное узкое место и сохраняет прозрачный файловый интерфейс для ML-runtime.

## 15. План формул

Формулы, которые стоит включить в основной текст:

1. Разложение cold start:
   `T_start = T_schedule + T_image_pull + T_unpack + T_init + T_model_download + T_model_load`.
2. AMAT для многоуровневого кэша:
   `AMAT = T_hit + (1 - H) * T_miss`.
3. Расширенная latency-модель:
   `E[T_read] = H_local*T_local + (1-H_local)*H_p2p*T_p2p + (1-H_local)*(1-H_p2p)*T_remote`.
4. Закон Литтла:
   `L = λW`.
5. Вероятность cold start:
   `P_cold = e^{-λT_keep_alive}`.
6. Zipf:
   `P(k) = k^{-α} / Σ i^{-α}`.
7. Che approximation:
   `h(n) = 1 - e^{-λ_n t_C}`.
8. Gossip convergence:
   `T = O(log N)`.
9. P2P download time:
   `T_download = S / (B_local + ΣB_i) + RTT`.
10. Replication availability:
    `A = 1 - (1-a)^r`.
11. Optimal chunk size:
    `C_opt = argmin_C(...)`.
12. Amdahl:
    `S(n) = 1 / ((1-p) + p/n)`.
13. BDP:
    `BDP = B * RTT`.
