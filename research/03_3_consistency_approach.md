# Подход к обеспечению консистентности при параллельном скачивании ML-моделей несколькими репликами

## 1. Введение: Проблема консистентности при параллельной загрузке моделей ML

В современных распределённых системах, особенно при работе с моделями машинного обучения (ML), часто возникает задача эффективной и консистентной доставки больших объёмов данных, таких как сами ML-модели, до множества вычислительных узлов. Представим сценарий, где N подов (например, в среде Kubernetes) одновременно стартуют и пытаются скачать одну и ту же ML-модель размером 70+ ГБ в общую директорию, смонтированную по протоколу NFS (Network File System). В такой ситуации критически важно обеспечить, чтобы только один под выполнял ресурсоёмкую операцию скачивания, в то время как остальные поды ожидали бы её завершения и затем использовали уже скачанные данные. Неконтролируемое параллельное скачивание не только приведёт к избыточной нагрузке на сеть и источник данных, но и может вызвать состояние гонки (race condition), приводящее к повреждению данных или неконсистентному состоянию файловой системы.

Целью данного документа является описание подхода к решению этой проблемы, основанного на механизмах файловых блокировок и атомарных операций с файлами. Будет рассмотрен алгоритм, позволяющий координировать действия подов, минимизировать накладные расходы и гарантировать целостность скачиваемых данных.

## 2. Механизмы файловых блокировок в Unix-подобных системах

Для координации доступа к общим ресурсам в Unix-подобных операционных системах традиционно используются файловые блокировки. Существует два основных типа системных вызовов для управления блокировками файлов: `flock()` и `fcntl()`.

### 2.1. `flock()`

Системный вызов `flock()` (от *file lock*) предоставляет консультативные (advisory) блокировки на уровне всего файла. Это означает, что операционная система не принуждает процессы соблюдать блокировки; вместо этого, корректно написанные приложения должны явно проверять и соблюдать их. `flock()` поддерживает два основных типа блокировок:

*   `LOCK_SH` (Shared Lock): Разделяемая блокировка. Позволяет нескольким процессам одновременно удерживать блокировку на одном файле. Обычно используется для операций чтения. Если файл уже заблокирован эксклюзивной блокировкой, запрос `LOCK_SH` будет ожидать её снятия.
*   `LOCK_EX` (Exclusive Lock): Эксклюзивная блокировка. Позволяет только одному процессу удерживать блокировку на файле. Используется для операций записи. Если файл уже заблокирован (разделяемой или эксклюзивной блокировкой), запрос `LOCK_EX` будет ожидать её снятия.

Дополнительно, флаг `LOCK_NB` (Non-Blocking) может быть объединён с `LOCK_SH` или `LOCK_EX` с помощью побитовой операции ИЛИ. При использовании `LOCK_NB` системный вызов `flock()` не будет блокировать выполнение процесса, а немедленно вернёт ошибку, если блокировка не может быть получена. Это позволяет реализовать неблокирующие попытки захвата блокировки и избежать зависания процесса.

Важной особенностью `flock()` является то, что блокировки привязаны к открытому файловому дескриптору и автоматически снимаются при его закрытии или завершении процесса, что предотвращает «мёртвые» блокировки в случае аварийного завершения приложения [3].

### 2.2. `fcntl()`

Системный вызов `fcntl()` (от *file control*) предоставляет более мощный и гибкий механизм блокировок, известный как байт-диапазонные блокировки (byte-range locks). В отличие от `flock()`, `fcntl()` позволяет блокировать не весь файл, а определённый диапазон байтов внутри него. Эти блокировки также являются консультативными, но могут быть сделаны принудительными (mandatory) в некоторых системах, хотя это редко используется из-за сложности и потенциальных проблем с производительностью. `fcntl()` также поддерживает разделяемые (`F_RDLCK`) и эксклюзивные (`F_WRLCK`) блокировки, а также неблокирующий режим (`F_SETLK`) [4].

## 3. Особенности файловых блокировок на NFS

Использование файловых блокировок в распределённых файловых системах, таких как NFS, сопряжено с определёнными сложностями и нюансами. Исторически `flock()` не всегда надёжно работал через NFS, поскольку его реализация часто была локальной для клиента, не распространяясь на сервер NFS. Однако, начиная с Linux 2.6.12, NFS-клиенты стали эмулировать `flock()` как байт-диапазонные блокировки `fcntl()` на всём файле, что позволило им работать через NFS [1]. Тем не менее, существуют важные аспекты, которые необходимо учитывать:

*   **Версия NFS**: Надёжная работа распределённых блокировок лучше поддерживается в NFSv4 по сравнению с NFSv3, где для управления блокировками использовались побочные протоколы (Network Lock Manager, NLM), которые могли быть источником проблем [1].
*   **Опции монтирования**: Для корректной работы распределённых блокировок на NFS критически важна опция монтирования `local_lock=none`. Эта опция указывает клиенту NFS не использовать локальные блокировки, а полностью полагаться на сервер NFS для управления ими. Если эта опция не установлена, клиент может применять локальные блокировки, которые не будут видны другим клиентам, что приведёт к нарушению консистентности [1].
*   **Стойкость к сбоям**: Распределённые блокировки на NFS должны быть устойчивы к сбоям клиентов. Сервер NFS должен иметь механизм для обнаружения «мёртвых» блокировок (например, от зависших клиентов) и их автоматического снятия. В NFSv4 это реализовано через механизм аренды (leases) и восстановления блокировок (lock reclamation) [5].

Несмотря на улучшения, `fcntl()` часто считается более надёжным выбором для распределённых блокировок на NFS, поскольку он был разработан с учётом сетевых файловых систем. Однако, поскольку в задании явно указан `flock`, мы будем использовать его, подразумевая, что среда NFS настроена для его корректной работы (например, с `local_lock=none` и достаточно современной версией NFS).

## 4. Предлагаемый алгоритм обеспечения консистентности

Для обеспечения консистентности при параллельном скачивании ML-модели несколькими подами в общую директорию NFS предлагается следующий алгоритм, использующий файловые блокировки и контрольные файлы-маркеры.

### 4.1. Структура директории и файлов

Предположим, что ML-модель `model.bin` должна быть скачана в директорию `/mnt/nfs/models/my_model/`. Для управления процессом скачивания будут использоваться следующие файлы:

*   `/mnt/nfs/models/my_model/model_download.lock`: Файл-мьютекс для управления доступом к операции скачивания. На него будут устанавливаться `flock` блокировки.
*   `/mnt/nfs/models/my_model/model.bin.partial`: Временный файл, в который будет скачиваться модель. Это позволяет избежать использования неполных или повреждённых данных.
*   `/mnt/nfs/models/my_model/.complete`: Файл-маркер, сигнализирующий о полном и успешном завершении скачивания модели.

### 4.2. Последовательность действий пода

Каждый под, которому требуется ML-модель, будет выполнять следующую последовательность действий:

1.  **Проверка наличия маркера завершения:**
    *   Перед попыткой скачивания, под проверяет наличие файла `/mnt/nfs/models/my_model/.complete`. 
    *   Если файл существует, это означает, что модель уже успешно скачана и готова к использованию. Под может сразу перейти к загрузке модели в память и начать работу.

2.  **Попытка захвата эксклюзивной блокировки (Лидер):**
    *   Если `.complete` файл отсутствует, под открывает файл `/mnt/nfs/models/my_model/model_download.lock` и пытается установить на него эксклюзивную блокировку (`LOCK_EX | LOCK_NB`).
    *   **Если блокировка успешно получена:** Этот под становится «лидером» и ответственным за скачивание модели. Он переходит к шагу 3.
    *   **Если блокировка не удалось получить (возвращена ошибка `EWOULDBLOCK` или аналогичная):** Это означает, что другой под уже захватил эксклюзивную блокировку и является лидером. Текущий под переходит к шагу 4.

3.  **Скачивание модели (Только лидер):**
    *   Лидер начинает скачивание ML-модели в временный файл `/mnt/nfs/models/my_model/model.bin.partial`.
    *   В процессе скачивания лидер может периодически проверять состояние блокировки, чтобы убедиться, что он всё ещё является единственным активным загрузчиком.
    *   После успешного завершения скачивания, лидер атомарно переименовывает `/mnt/nfs/models/my_model/model.bin.partial` в `/mnt/nfs/models/my_model/model.bin`.
    *   Затем лидер атомарно создаёт файл-маркер `/mnt/nfs/models/my_model/.complete`. Атомарность создания маркера критически важна для предотвращения ситуаций, когда другие поды могут начать использовать неполную модель.
    *   После создания маркера лидер снимает эксклюзивную блокировку с `model_download.lock`.

4.  **Ожидание завершения скачивания (Ожидающие поды):**
    *   Поды, которым не удалось получить эксклюзивную блокировку, открывают файл `model_download.lock` и пытаются установить на него разделяемую блокировку (`LOCK_SH`).
    *   Запрос `LOCK_SH` будет блокирующим, то есть под будет ожидать, пока лидер не снимет свою `LOCK_EX` блокировку. Это гарантирует, что ожидающие поды не будут потреблять ресурсы CPU в цикле активного ожидания.
    *   Как только разделяемая блокировка получена, это означает, что лидер завершил свою работу и снял эксклюзивную блокировку. Ожидающий под снимает разделяемую блокировку.
    *   После снятия разделяемой блокировки, ожидающий под переходит к шагу 1 (повторная проверка файла-маркера `.complete`), чтобы убедиться, что модель действительно готова. Это необходимо, так как снятие `LOCK_EX` может произойти по разным причинам (например, сбой лидера), и повторная проверка маркера гарантирует использование только полностью скачанной модели.

## 5. Предотвращение состояний гонки (Race Conditions)

Описанный алгоритм специально разработан для минимизации и предотвращения состояний гонки. Рассмотрим потенциальные race conditions и способы их избежания:

*   **Два пода одновременно начинают скачивание:** Это предотвращается использованием `LOCK_EX | LOCK_NB`. Только один под сможет успешно захватить эксклюзивную блокировку. Остальные будут немедленно проинформированы о занятости блокировки и перейдут в режим ожидания.
*   **Под начинает использовать неполную модель:** Это предотвращается двумя механизмами:
    *   Скачивание всегда происходит во временный файл (`.partial`).
    *   Модель становится доступной для использования только после атомарного переименования временного файла в окончательное имя (`model.bin`) и атомарного создания файла-маркера (`.complete`). Операция `os.Rename` (или `mv` в POSIX-системах) является атомарной в пределах одной файловой системы, что гарантирует, что другие процессы увидят либо старую версию файла, либо новую, но никогда не промежуточное состояние. Создание файла-маркера также должно быть атомарным (например, `os.Create` в Go, если файл не существует, или `touch` в shell).
*   **Сбой лидера во время скачивания:** Если под-лидер завершается аварийно до завершения скачивания и создания файла `.complete`, его эксклюзивная блокировка будет автоматически снята операционной системой (благодаря свойству `flock` привязываться к файловому дескриптору процесса). В этом случае один из ожидающих подов сможет захватить эксклюзивную блокировку и стать новым лидером, начав скачивание заново. Важно, что временный файл `.partial` не будет использоваться, так как он не был атомарно переименован и не сопровождался файлом `.complete`.
*   **Неконсистентное состояние после сбоя:** Использование файла-маркера `.complete` является ключевым. Даже если блокировка была снята, ожидающие поды всегда проверяют наличие этого маркера. Если его нет, они не будут использовать потенциально повреждённые данные и будут ждать нового лидера или сами станут им.

```pseudocode
            ReleaseLock(lockFile)
            Log("Модель уже скачана другим подом, используем существующую.")
            return SUCCESS

            ReleaseLock(lockFile)
            return SUCCESS

            Log("Ошибка скачивания модели.")
            ReleaseLock(lockFile)
            DeleteFile(partialFilePath) // Удаляем неполный файл
```

### 6.2. Пример Go-кода

Для реализации `flock` в Go можно использовать пакет `syscall` или сторонние библиотеки, такие как `github.com/gofrs/flock`, которая предоставляет более удобный API. В данном примере будет показан подход с `syscall` для наглядности.

```go
package main

import (
	"fmt"
	"io"
	"io/ioutil"
	"log"
	"os"
	"path/filepath"
	"syscall"
	"time"
)

const (
	modelName       = "my_ml_model.bin"
	modelSize       = 70 * 1024 * 1024 * 1024 // 70 GB
	modelSourceURL  = "http://example.com/large_model.bin" // Заглушка
	baseDir         = "/mnt/nfs/models/my_model"
	lockFileName    = "model_download.lock"
	completeFileName = ".complete"
	partialFileName = "model_download.partial"
)

func main() {
	if err := os.MkdirAll(baseDir, 0755); err != nil {
		log.Fatalf("Не удалось создать директорию %s: %v", baseDir, err)
	}

	modelPath := filepath.Join(baseDir, modelName)
	lockFilePath := filepath.Join(baseDir, lockFileName)
	completeFilePath := filepath.Join(baseDir, completeFileName)
	partialFilePath := filepath.Join(baseDir, partialFileName)

	log.Printf("Попытка загрузки модели: %s", modelPath)

	for {
		if isModelComplete(completeFilePath) {
			log.Printf("Модель %s уже скачана. Завершение.", modelName)
			return
		}

		// Попытка захвата эксклюзивной блокировки
		lockFile, err := os.OpenFile(lockFilePath, os.O_CREATE|os.O_RDWR, 0644)
		if err != nil {
			log.Printf("Ошибка открытия файла блокировки %s: %v. Повторная попытка через 5 секунд.", lockFilePath, err)
			time.Sleep(5 * time.Second)
			continue
		}

		err = syscall.Flock(lockFile.Fd(), syscall.LOCK_EX|syscall.LOCK_NB)
		if err == nil {
			// Мы - лидер, захватили эксклюзивную блокировку
			log.Printf("Захвачена эксклюзивная блокировка. Начинаем скачивание %s.", modelName)

			// Повторная проверка маркера на случай, если другой под успел завершить работу
			if isModelComplete(completeFilePath) {
				log.Printf("Модель %s уже скачана другим подом. Снимаем блокировку и завершаем.", modelName)
				syscall.Flock(lockFile.Fd(), syscall.LOCK_UN)
				lockFile.Close()
				return
			}

			err = downloadModel(modelSourceURL, partialFilePath, modelSize)
			if err != nil {
				log.Printf("Ошибка скачивания модели %s: %v. Снимаем блокировку.", modelName, err)
				syscall.Flock(lockFile.Fd(), syscall.LOCK_UN)
				lockFile.Close()
				os.Remove(partialFilePath) // Удаляем неполный файл
				time.Sleep(10 * time.Second) // Ждем перед повторной попыткой
				continue
			}

			// Атомарное переименование и создание маркера
			if err := os.Rename(partialFilePath, modelPath); err != nil {
				log.Printf("Ошибка атомарного переименования %s в %s: %v. Снимаем блокировку.", partialFilePath, modelPath, err)
				syscall.Flock(lockFile.Fd(), syscall.LOCK_UN)
				lockFile.Close()
				time.Sleep(10 * time.Second)
				continue
			}

			// Создание файла-маркера
			_, err = os.Create(completeFilePath)
			if err != nil {
				log.Printf("Ошибка создания файла-маркера %s: %v. Снимаем блокировку.", completeFilePath, err)
				syscall.Flock(lockFile.Fd(), syscall.LOCK_UN)
				lockFile.Close()
				time.Sleep(10 * time.Second)
				continue
			}

			log.Printf("Модель %s успешно скачана и маркер создан. Снимаем блокировку.", modelName)
			syscall.Flock(lockFile.Fd(), syscall.LOCK_UN)
			lockFile.Close()
			return
		} else if err == syscall.EWOULDBLOCK {
			// Эксклюзивная блокировка занята, ждем
			log.Printf("Эксклюзивная блокировка занята. Ожидаем завершения скачивания %s.", modelName)
			err = syscall.Flock(lockFile.Fd(), syscall.LOCK_SH) // Блокирующее ожидание
			if err != nil {
				log.Printf("Ошибка при ожидании разделяемой блокировки: %v. Повторная попытка через 5 секунд.", err)
				lockFile.Close()
				time.Sleep(5 * time.Second)
				continue
			}
			log.Printf("Разделяемая блокировка получена. Снимаем и проверяем маркер.")
			syscall.Flock(lockFile.Fd(), syscall.LOCK_UN)
			lockFile.Close()
			// После снятия блокировки, снова проверяем маркер
			if isModelComplete(completeFilePath) {
				log.Printf("Модель %s успешно скачана другим подом. Завершение.", modelName)
				return
			} else {
				log.Printf("Маркер %s не найден после ожидания. Возможно, лидер завершился с ошибкой. Повторяем попытку.", completeFileName)
				time.Sleep(5 * time.Second)
				continue
			}
		} else {
			log.Printf("Неизвестная ошибка при попытке блокировки: %v. Повторная попытка через 5 секунд.", err)
			lockFile.Close()
			time.Sleep(5 * time.Second)
			continue
		}
	}
}

func isModelComplete(completeFilePath string) bool {
	_, err := os.Stat(completeFilePath)
	return !os.IsNotExist(err)
}

func downloadModel(url, destPath string, size int64) error {
	log.Printf("Начинается имитация скачивания %s в %s (размер: %d байт)", url, destPath, size)
	// Имитация скачивания большого файла
	file, err := os.Create(destPath)
	if err != nil {
		return fmt.Errorf("не удалось создать временный файл: %w", err)
	}
	defer file.Close()

	// Имитация записи данных
	buffer := make([]byte, 1024*1024) // 1 MB буфер
	for i := 0; i < int(size/(1024*1024)); i++ {
		_, err := file.Write(buffer)
		if err != nil {
			return fmt.Errorf("ошибка записи в временный файл: %w", err)
		}
		// log.Printf("Скачано %d MB...", (i+1))
		time.Sleep(50 * time.Millisecond) // Имитация сетевой задержки
	}
	log.Printf("Имитация скачивания %s завершена.", url)
	return nil
}
```

## 7. Заключение

Предложенный подход, основанный на использовании файловых блокировок `flock` и атомарных операций с файлами в общей директории NFS, позволяет эффективно решать проблему консистентной загрузки больших ML-моделей несколькими параллельно стартующими подами. Ключевыми элементами являются:

*   **Эксклюзивная блокировка (`LOCK_EX|LOCK_NB`)** для выбора одного лидера, ответственного за скачивание.
*   **Разделяемая блокировка (`LOCK_SH`)** для блокирующего ожидания остальными подами.
*   **Временный файл (`.partial`)** для предотвращения использования неполных данных.
*   **Атомарное переименование** для публикации готовой модели.
*   **Файл-маркер (`.complete`)** для сигнализации о полном и успешном завершении процесса.

Такой механизм обеспечивает надёжность и высокую доступность, минимизируя сетевой трафик и нагрузку на источник данных, а также предотвращая состояния гонки, которые могут привести к повреждению данных. Важно учитывать особенности работы `flock` на NFS и убедиться, что файловая система настроена для корректной поддержки распределённых блокировок.
